# ChemVCS 用户访谈脚本与研究计划

> v1.0 | 2026-02-09  
> 执行周期：2 周（2026-02-10 至 2026-02-23）  
> 目的：验证 P0 痛点真实性、确认 MVP 范围、识别采用阻力

---

## 0. 待验证核心假设总表

在访谈开始前，团队需对齐以下假设——每一条都必须在 2 周内得到"成立/不成立/需修改"的判定。

| ID | 假设 | 类型 | 判定成立的最低门槛 |
|----|------|------|-------------------|
| H1 | 计算材料研究者存在严重的版本混乱问题（P0 级） | 痛点 | ≥80% 受访者在过去 3 个月内至少 1 次因找不到参数而浪费 ≥1h |
| H2 | 现有解决方案（手动目录/Git/AiiDA）未解决版本追踪需求 | 竞品 | ≥70% 受访者未使用任何专用版本管理，或使用后仍有显著不满 |
| H3 | 用户愿意在计算流程中额外执行 1-2 条 CLI 命令 | 采用 | ≥60% 受访者表示可接受 ≤30s 的额外操作 |
| H4 | "语义 diff"（参数级对比而非文本行对比）是有吸引力的差异化功能 | 价值 | ≥70% 受访者在看到 diff 示例后表示"明显优于现有方式" |
| H5 | 博士生是正确的楔子用户（而非博后或 PI） | 市场 | 博士生组痛点频率和强度 ≥ 博后组 |
| H6 | pip install 是可接受的安装方式，无需 root 是硬约束 | 环境 | ≥90% 超算环境允许 pip install --user |
| H7 | 赝势文件管理（多副本、版本不确定）是真实子痛点 | 痛点 | ≥50% 受访者提及赝势相关困惑 |
| H8 | 学生毕业交接导致计算项目断层是 PI 关心的问题 | 痛点 | 博后组 ≥2 人有交接断层的亲身经历 |
| H9 | 用户愿意投入 ≤15 分钟将现有项目迁移至新工具 | 采用 | ≥60% 受访者表示 15 分钟可接受 |
| H10 | 共享文件系统（Lustre/GPFS）上无额外服务依赖是硬需求 | 技术 | 运维组 100% 确认不接受 PostgreSQL/MongoDB 级别依赖 |

## 1. 访谈对象分组

### Group A：博士研究生（目标 8 人，最少 6 人）

| 项目 | 内容 |
|------|------|
| **筛选条件** | ①在读硕/博士生 ②使用 VASP 进行第一性原理计算 ③在超算（SLURM 或 PBS）上提交任务 ④至少有 6 个月的 VASP 使用经验 ⑤排除：已深度使用 AiiDA/atomate2 的"高级用户" |
| **目标** | 验证 P0 痛点（版本混乱、复现困难）的真实性和严重程度；了解当前记录习惯；测试语义 diff 的感知价值；探测采用阻力 |
| **关键假设** | H1（版本混乱是 P0）、H2（现有方案不足）、H3（愿意多敲命令）、H4（语义 diff 有价值）、H5（博士生是楔子）、H9（迁移成本可接受） |
| **成功判据** | ≥6/8 人的痛点叙述中自发提及参数记录/版本查找问题；≥5/8 人对语义 diff 示例给出正面评价（"有用"/"比我现在的方法好"） |

### Group B：博后 / 助理研究员（目标 3 人，最少 2 人）

| 项目 | 内容 |
|------|------|
| **筛选条件** | ①博后或助理研究员 ②具有 ≥2 年计算材料经验 ③管理过多人协作的计算项目或高通量筛选 ④Git 熟练用户 |
| **目标** | 对比博士生组的痛点强度；了解协作/交接场景的真实摩擦；识别 MVP 之外的高优需求（branch、批量操作）；作为"高级用户"检验 CLI 设计的合理性 |
| **关键假设** | H5（博士生 vs 博后谁痛更甚）、H8（交接断层）、H2（AiiDA/atomate2 是否已解决问题） |
| **成功判据** | ≥2/3 人曾经历或目睹项目交接断层；能明确描述现有工具链的不足之处 |

### Group D：超算中心运维 / 平台工程师（目标 2 人，最少 1 人）

| 项目 | 内容 |
|------|------|
| **筛选条件** | ①在高校/研究所超算中心工作 ②负责用户支持或存储管理 ③了解 Lmod/EasyBuild 及共享文件系统 |
| **目标** | 验证 HPC 环境技术约束；了解用户存储问题的运维视角；评估 ChemVCS 的部署可行性（pip、module、文件系统锁） |
| **关键假设** | H6（pip install 可行）、H10（无额外服务依赖是硬需求）、技术假设（SQLite 在共享 FS 上的行为） |
| **成功判据** | 确认 pip install --user 在其集群上可行；确认不接受额外数据库/守护进程；获得对 SQLite-on-Lustre 的实操经验或意见 |

## 2. 访谈问题清单

> **通用说明：** 每个问题从开放式提问开始，避免引导。标注 `[深挖]` 的是追问方向，在受访者给出初步回答后使用。标注 `[展示]` 的需要出示辅助材料（见附录 A）。

### 2.1 Group A 问题清单（博士生，~45 min）

#### 热身与背景（5 min）

**A01.** 能简单介绍一下你目前的研究课题和使用的计算方法吗？
- 目的：建立上下文，确认符合筛选条件
- `[深挖]` 你主要用 VASP 的哪些功能？（弛豫/电子结构/声子/NEB...）

**A02.** 你在超算上大概有多少个正在进行或已完成的计算项目？
- 目的：量化规模，判断版本管理需求的基础
- `[深挖]` 每个项目通常包含多少次独立的计算（VASP 运行）？

#### 版本管理现状（10 min）

**A03.** 你现在是怎么组织超算上的计算目录的？能描述一下你的命名规则吗？
- 目的：**验证 H1** — 版本混乱的程度
- `[深挖]` 有没有遇到过找不到某个计算用了什么参数的情况？最近一次是什么时候？花了多长时间？

**A04.** 当你修改了 INCAR 中的某个参数（比如 ENCUT 或 Hubbard U），你怎么记录这次修改？
- 目的：**验证 H1** — 了解现有记录手段
- `[深挖]` 你会在哪里记下"为什么要改这个参数"？（纸质笔记本？txt 文件？什么都不记？）

**A05.** 你有没有用 Git 来管理计算文件？为什么用/为什么不用？
- 目的：**验证 H2** — 现有方案的采用情况
- `[深挖]` 如果用了：Git 在管理 VASP 文件时有什么让你不满意的地方？如果没用：你觉得主要障碍是什么？

**A06.** 你听说过或尝试过 AiiDA、signac、atomate2 吗？体验如何？
- 目的：**验证 H2** — 竞品认知和使用情况
- `[深挖]` 如果尝试过：为什么继续用/放弃了？

#### 复现与回溯（8 min）

**A07.** 在过去半年里，你有没有需要回去找"以前某次计算的精确参数"的经历？
- 目的：**验证 H1** — 回溯需求的频率
- `[深挖]` 那次经历的结果如何——找到了还是没找到？花了多长时间？是什么场景（论文撰写？导师提问？审稿人要求？）

**A08.** 如果论文审稿人要求你证明某个计算结果的可复现性，你现在的准备程度如何？（1-5 分）
- 目的：**验证 H1** — 复现困难的自评
- `[深挖]` 你觉得最大的障碍是什么？（参数丢失？赝势不确定？VASP 版本不同？）

**A09.** 你在做计算时会记录使用的赝势文件版本吗？
- 目的：**验证 H7** — 赝势管理子痛点
- `[深挖]` 你组里是怎么管理赝势文件的？（共享目录？各自拷贝？你知道你现在用的是哪个版本吗？）

#### 概念测试与价值探测（12 min）

**A10.** `[展示]` 我给你看一个假想的工具输出。这是两次计算之间的"参数对比"。你觉得这种信息对你有用吗？
- 目的：**验证 H4** — 语义 diff 的感知价值
- 展示材料：附录 A 中的"语义 diff 示例"
- `[深挖]` 这和你现在的做法（手动比较 INCAR 文件）相比，差别有多大？（无差别 / 有点用 / 明显有用 / 必须有）

**A11.** 假设有这样一个工具：在你每次改完参数后，敲一条命令就能自动记录。你愿意使用吗？
- 目的：**验证 H3** — 额外操作的接受度
- `[深挖]` 你觉得这条命令每次花多少秒是可以接受的？如果需要两条命令呢（`add` + `commit`）？

**A12.** 如果你现在已经有 50 个计算目录了，把它们迁移到新工具里，你愿意花多长时间？
- 目的：**验证 H9** — 迁移成本阈值
- `[深挖]` 每个目录花 5 分钟？15 分钟？30 分钟？什么时候你会放弃？

#### 反证与阻力（5 min）

**A13.** 什么情况下你**绝对不会用**这类工具？
- 目的：识别采用阻力和 deal-breaker
- `[深挖]` 如果这个工具需要你先学 30 分钟呢？如果要装额外的数据库呢？如果你的同事都不用呢？

**A14.** 假设你试用了这个工具两周，什么情况下你会停止使用？
- 目的：识别留存风险
- `[深挖]` 速度太慢？功能太少？跟你的习惯冲突？

#### 收尾（5 min）

**A15.** 在你的日常计算工作中，如果有一个工具能帮你解决**一个**问题，你最希望它解决什么？
- 目的：开放探测——确认 P0 痛点排序是否正确
- 不做引导，记录原话

**A16.** 你愿意参与后续的原型测试吗？如果工具做出来了，你愿意在真实项目上试用多久来评估？
- 目的：确认后续测试的样本可用性

## 3. 任务式验证（所有 Group A/B 受访者执行）

> **说明：** 以下任务在访谈中段进行（问完版本管理现状之后）。请受访者共享屏幕或现场演示。每个任务限时 5 分钟。记录员重点观察**障碍、犹豫、自言自语、求助行为**。

### Task 1：展示你的目录结构（5 min）

**指令：** "请打开你在超算上某个计算项目的目录，给我们看看它的结构。请一边浏览一边说明你的命名规则。"

**观察要点：**

| 观察维度 | 记录内容 |
|----------|----------|
| 目录嵌套深度 | 几层？命名是否可理解？ |
| 版本标识方式 | v1/v2？final/真final？日期？ |
| 参数记录位置 | 有无 README / notes.txt / log？ |
| 赝势文件位置 | 统一路径？多处拷贝？ |
| 输出文件处理 | WAVECAR 是否残留？占多少空间？ |
| 受访者情绪 | 是否表现出尴尬/困惑/解释困难？ |

### Task 2：复现一个旧结果（5 min）

**指令：** "假设你的导师现在问你：'你上个月做的 [某个具体体系] 那组计算，用的什么参数？结果是多少？'请你现在尝试找到它。"

**观察要点：**

| 观察维度 | 记录内容 |
|----------|----------|
| 查找策略 | ls / find / grep / 凭记忆？ |
| 所需时间 | 从开始到找到（或放弃），计时 |
| 准确性 | 找到的参数确实是那次计算的吗？能确认吗？ |
| 口头表达 | "我记得应该在这个目录..." / "不确定是不是这个版本..." |
| 赝势确认 | 能否确认当时用的赝势版本？ |
| 失败点 | 在哪一步卡住了？ |

### Task 3：参数对比（3 min，仅限找到 ≥2 个版本的受访者）

**指令：** "你能比较一下你刚才找到的计算和另一个版本之间，INCAR 参数有什么不同吗？"

**观察要点：**

| 观察维度 | 记录内容 |
|----------|----------|
| 使用工具 | diff / vimdiff / 肉眼？ |
| 比较范围 | 只比 INCAR？还是也比 POSCAR/KPOINTS？ |
| 遗漏参数 | 是否有遗漏的差异未被发现？ |
| 所需时间 | 计时 |
| 表达 | "这太麻烦了" / "我一般不做这种比较" |

## 4. 反证问题（Disconfirmation）

> 以下问题穿插在各组访谈中，不单独成为一个环节。目的是主动寻找"我们可能是错的"的证据。

| # | 反证问题 | 适用组 | 如果答案是... | 则意味着... |
|---|----------|--------|---------------|-------------|
| DC1 | "你觉得你现在的方法虽然不完美，但基本够用吗？" | A/B | "够用了" ≥50% | H1 痛点可能被高估；需要下调优先级或重新定义价值主张 |
| DC2 | "如果 Git 加一个 VASP 插件就行了，你还需要新工具吗？" | A/B | "Git 插件就够了" ≥40% | 需重新评估是否做独立工具 vs Git 插件 |
| DC3 | "你的组里有没有人因为用了 AiiDA 或类似工具，已经解决了这些问题？" | A/B | ≥2 人确认 AiiDA 已够用 | 竞品假设 H2 可能不成立 |
| DC4 | "如果这个工具只能在你的本地电脑上跑，不能在超算上用，你还用吗？" | A | "不用" ≥70% | 确认 HPC 原生是硬需求 |
| DC5 | "如果学会使用需要 2 小时的教程，你愿意花这个时间吗？" | A | "不愿意" ≥50% | 学习曲线上限 <2h；MVP 必须做到 ≤30 min 上手 |
| DC6 | "你组里有没有人完全不用命令行的？他们怎么做计算？" | A/B | "有，他们用 GUI/脚本包装" | 存在非 CLI 用户群体，可能需要不同的入口 |
| DC7 | "如果我们把文件系统存储换成需要你跑一个数据库服务的方案，你的反应是？" | D | "绝对不行" vs "可以考虑" | 验证 H10 的硬度 |

## 5. 价格 / 采用门槛探测

在 A/B 组访谈的概念测试环节中使用：

```

   选项：<10 秒 / <30 秒 / <1 分钟 / <5 分钟
```

```
   答案记录：有网络 / 需要代理 / 离线安装 wheel / 找运维帮忙
```

## 9. 分析框架

### 9.1 假设板（Assumption Board）

```
                    高影响
                      │
         ┌────────────┼────────────┐
         │  H1 版本混乱 │ H3 愿意多敲 │
 高风险  │  H4 语义diff │ H5 楔子用户 │
（不确定）│  H8 交接断层 │ H9 迁移成本 │
         ├────────────┼────────────┤
         │  H7 赝势管理 │ H6 pip可用  │
 低风险  │             │ H10 无依赖  │
（有信心）│             │ H2 方案不足 │
         └────────────┼────────────┘
                      │
                    低影响
```

### 9.2 证据强度评定规则

| 等级 | 定义 |
|------|------|
| **强** | 受访者自发提及该痛点/需求（未经引导）；有具体的时间/事件/数字佐证 |
| **中** | 受访者在追问下确认该痛点存在；能给出模糊的例子 |
| **弱** | 受访者在直接提问后表示"好像有这个问题"但无法给出实例 |
| **反证** | 受访者明确否定该假设，或给出反面证据 |
